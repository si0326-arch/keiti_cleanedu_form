# 작업종료 마무리 점검 (wrap-up)

사용자가 **"작업종료"** 라고 하면(또는 `/wrap-up`), 컴퓨터를 끄기 전에 아래 3가지를 점검하고 결과를 표로 요약한다.
**점검만 한다 — 커밋·푸시·DB변경 같은 행위는 사용자 확인(또는 "저장해줘") 후에만 한다.**

## 1. GitHub (커밋·푸시)
- `git status --short` → 커밋 안 된 변경이 있는지.
- `git rev-list --left-right --count origin/main...main` → 로컬이 원격보다 앞서거나(미푸시 커밋) 뒤처졌는지.
- 미커밋 변경 또는 미푸시 커밋이 있으면 → **목록을 보여주고 저장할지 묻는다.**
  - 이 프로젝트 규칙: 사용자가 "저장해줘"라고 하면 `git add -A` → commit → `git push origin main`.

## 2. Vercel (배포)
- 라이브 응답 확인: `curl -sL -o /dev/null -w "HTTP %{http_code}\n" https://keiti-cleanedu-form.vercel.app/`
- 방금 main에 push했다면 자동배포에 1~2분 걸릴 수 있음 → 진행 중이면 그렇게 안내.
- (선택) index.html의 `/assets/*.js` 번들을 받아 최신 코드(기대 문자열) 반영 여부 확인.

## 3. Supabase (DB)
- `supabase/migrations/` 의 `.sql` 파일이 모두 git에 커밋됐는지 (1번 점검에 포함됨).
- 이번 세션에서 스키마/데이터를 바꿨다면 → **마이그레이션 파일로 만들어 실행했는지** 확인 (CLAUDE.md 규칙: 즉석 변경 금지, 롤백 가능하게).
- 비밀 파일이 새지 않았는지: `git check-ignore .env.migration` (출력되면 = 무시됨 = 정상).
- 남아있는 테스트 행이 있으면 "정리할지" 한 번 언급 (삭제는 마이그레이션 파일로).

## 4. 오늘의 작업 요약 (체크 후 — 사용자가 이걸 보고 컴퓨터를 끔)
- 위 점검을 마친 뒤, **이번 세션에서 한 작업을 사용자가 이해하기 쉬운 말로 요약**해서 설명한다.
- 포함: 무엇을 바꿨는지 / 무엇이 배포·DB에 반영됐는지 / 지금 시스템 상태 / 남은 일(있으면).
- 전문용어는 풀어서, 항목별로 짧고 또렷하게. 사용자는 이 요약을 보고 안심하고 컴퓨터를 끈다.

## 출력 형식
- 각 항목을 **✅ 정상 / ⚠️ 조치 필요** 로 표시.
- ⚠️ 항목마다 "무엇을 하면 되는지" 한 줄 제시.
- 마지막 줄: 모두 ✅ 면 **"이제 꺼도 됩니다 👍"**, 아니면 **"끄기 전에 [X] 하세요"**.

## 주의
- 읽기/상태확인 위주로만 점검한다. 파일·DB·원격을 바꾸는 행위는 사용자 승인 후.
- 새 프로젝트에 이 스킬을 복사해 쓸 경우 Vercel URL 등 프로젝트별 값은 상황에 맞게 조정한다.
